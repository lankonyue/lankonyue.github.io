改掉这几个毛病，你的代码也会很优雅

发表于2025-12-09|更新于2025-12-13|通用编程
|总字数:1.8k|阅读时长:8分钟|浏览量:

最近1年，做了很多代码审核工作，团队中的成员有刚毕业的，也有久战沙场的。但不管是新人，还是老手，都会发现一些问题。有些英语不好，类名、变量名、方法名乱取；有些不注重代码性能，for 循环从不省着用；也有些喜欢 copy/paste, 同样的代码满天飞。还有很多各种各样的问题，就不一一例举了。

接下来，我就以Qt C++ 实现登陆模块的代码为例，通过最常见的 11 条反例和正例来说明。如果这11条都能做到，那你的代码给人的感觉肯定是赏心悦目，清爽利落。

注：非公司的实际项目代码，只是用来例举下常见问题

1. Bool 型变量取名规则

简单记忆口诀

检查状态用 is，拥有关系用 has
包含检查用 contains，能力判断用 can
需求建议用 should，存在检查用 exists
肯定形式是王道，否定逻辑用 ! 号

2. 直接使用拼音命名: 难理解

不要使用拼音，使用拼音，还不如直接使用 中文得了。所以还是要用英文。

3. 中式英语字面直译：别扭绕口

变量名、方法名要符合英文语义，要通顺、流畅

4. 名字上下文不一致性：混乱

一个事物，总是用相同的名字，这样不会造成理解偏差。就像两个人谈话，对同一个东西，使用相同的名词，这样两人沟通就更顺畅。

5. 函数形参个数太多：易弄错，不易扩展

一般情况下，形参个数达到 5个及以上时，就可以考虑使用结构体或类来包装。如果形参太多，还可以将形参按相关性进行分组，使用多个结构体或类来包装。

6. 类中存在多处重复代码：难维护

重复代码危害很大，比如有10处重复代码，恰好有 bug, 那就需要改 10处。如果提炼成一个方法供其它地方调用，只需要修复这一处即可。

7. if 嵌套层数太深：难阅读

一般情况下，if 层数控制在不超过 3 层。超过3层就想办法进行代码优化。

8. 非专有名词的变量乱缩写：理解烧脑

9. 执行无效代码：浪费

代码在需要它的时候再执行，按需执行，不要浪费。

10. 不知提前结束循环：性能损耗

写代码的原则是：能省CPU，就尽量省，哪怕几个CPU周期。

11. 函数代码行数太多：看到下面，忘记上面

一个函数代码行数太长，就会带来阅读上的困难。可以拆成多个小函数，这样就可以很清晰的看清这个大函数的结构脉络。
